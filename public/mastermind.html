About

Mastermind is a board game invented in 1970 by Mordecai Meirowitz, though it bears many similarities to an older British pen-and-paper game, Bulls and Cows1. The algorithmic study of Mastermind began in earnest in 1976 with Donald Knuth’s essay “The Computer as Master Mind”2, in which he outlined a strategy for consistently winning a match in under 5 guesses by always choosing a query that, in his words, “minimizes the maximum number of remaining possibilities.”2(pg.3) In the decades since, Mastermind has cropped up numerous times in computer science academia due to its relevance in a variety of applications. A common thread that joins the bulk of these inquiries, however, is a shared goal: to find a more optimal Mastermind strategy.

How to play

  1. This version of Mastermind is played with a secret-code that is 4 colors long, made from any combination of 6 available colors.

    • Repeating colors is permitted without restriction, but the code will always be 4 colors long (for example, the code 1-1-1-1 is a possibility, but 1-1-1 is not).

      • This program utilizes RANDOM.ORG's web API to generate the secret code for each game—ergo, duplicate colors within codes are rather common.

  2. In the graphical implementation of the game, you are the codebreaker.

  3. Form a guess by clicking and dragging the colored marbles on the left-hand side of the screen into the board in the middle. Alternatively, you may use the numbered keys 1-6 to quickly place marbles starting from the left.

    • You can rearrange the marbles before committing to a guess. To delete marbles you’ve already placed, drag them back over to the left-hand container and let go of the mouse button.

  4. Click the green ‘Check’ button or press Enter to finish your turn. Feedback will appear to the right of the play area.

  5. Feedback consists of black pegs and white pegs.

    • Black pegs are given for each marble that is the correct color and in the correct spot.

    • White pegs are given for each marble that is the correct color, but in an incorrect spot.

  6. Guess the code within 10 rounds to win the game.

Decision problem

The Mastermind Satisfiability Problem (or MSP) is an NP-complete decision problem first described by Stuckman & Zhang in their aptly-named 2006 paper “Mastermind is NP-complete”4:

Given a set of guesses , and their corresponding scores, is there at least one valid solution?

Here, is the set of natural numbers (1, 2, 3, ...), is the available colors, and is the length of the code sequence. Therefore the set of guesses is a subset of constrained by colors and length .

The inputs and outputs of MSP are given as follows:

In that same essay4, the NP-completeness of MSP is shown. Here’s a quick rundown of their proof:

  1. MSP NP

    Regarding the MSP’s polynomial-time verifiability (and thus membership in NP):

    It is apparent that the validity of a solution for an instance of MSP can be evaluated in polynomial time, because checking a satisfying peg configuration is as easy as matching the pegs against each guess.

  2. MSP NP-hard

    Proven via reduction from the NP-hard VERTEX-COVER( ), which is the problem of determining whether there exists a size subset of vertices in a graph where each edge is connected to at least one vertex in . This is accomplished by taking the VERTEX-COVER instance where and its graph and converting it to MSP by setting the colors equal to and the length equal to . From there, a set of guesses are made which establish the constraints along with two control answers and .

    It is then shown that if the MSP instance constructed in the last paragraph has a solution (or does not), then there exists a size- vertex-cover for . In addition, it is shown that if there exists a size- vertex-cover for , then there exists a solution to the MSP instance. Hence the if and only if relationship is established.

    The proof is finalized with the assertion that this reduction is “polynomial in input size”4.

Algorithms

Three algorithms were implemented for this project:

  1. User answer verification by comparing inputs to the solution key.
  2. Answering the decision question by processing the game’s inputs and outputs in a way that’s agnostic to whether the human and the computer followed the rules.
  3. A solution algorithm which plays the game and forms guesses based on the feedback from past answers.

Algorithm for decision question

A convenient property of Mastermind scores

Most of what follows in this section is dependent upon this crucial insight from DelphiForFun’s Gary Darby5:

Score(secret pattern, guess) = Score( guess, secret pattern). This implies that given any guess and the resulting score, we can pass through all eligible patterns and select those that produce the same score as the score we were given. The solution is guaranteed to be among these.

This property allowed me to implement a very straightforward ‘check’ mechanism: After each user input and the subsequent feedback, the list of candidate solutions will be pruned in order to exclude all answers which do not score the same against previous guesses as the previous guesses did against the secret pattern.

Relation of algorithm to coursework

This validation algorithm is a backtracking algorithm6, as it references a list of candidate solutions which is reduced at each turn. This is done by iterating through all remaining candidates and excising those that are found to be unacceptable with the current state of information. The reason this works is because the property mentioned earlier, Score(secret pattern, guess) = Score(guess, secret pattern), also implies that any valid candidate solution will score the same as a previous guess when scored against the previous guess as if that were the answer. In other words, the distance from the candidate solution to a past guess will be the same as the distance from the past guess to the answer. Otherwise the candidate solution is not valid and should be removed from the list.

Using this method, as long as the list of candidate solutions remains populated, then there is at least one valid solution to the Mastermind instance.

Making guesses

The method used for automatically generating new guesses is based on an algorithm described by Temporal & Kovacs in their work “A heuristic hill climbing algorithm for Mastermind”.7 This strategy also uses the fact that Score(secret pattern, guess) = Score(guess, secret pattern) to evaluate candidate solutions. The method for choosing a guess goes as follows7:

  1. Submit a random guess and call it CFG [Current Favorite Guess].
  2. Induce a New Potential Guess (NPG) from CFG [...]
  3. If NPG does not score the same as previous guesses go to (2), otherwise submit it as new guess.
  4. If new guess scores 0 black 0 white then prevent its colours being used again and go to (1).
  5. If new guess scores as good or better than CFG then adopt it as CFG.
  6. If new guess is a secret code stop, otherwise (2).

You can view this algorithm at work by clicking the “Pick a guess for me!” button in the GUI application, or by removing the comment (//) before serverSideGame() on the last line of app.js (which will have the program play 7 games on its own, in quick succession).
